# 随访分组管理模块 - 本周深度研发笔记 (2026-04-13 ~ 04-17)

## ✍️ 任务综述
本周核心任务是重构并优化“随访分组与周期管理”模块。该模块在业务上具有高度的关联性，技术上采用了非传统的**“父子同表（Single Table Inheritance）”**架构设计。本周不仅完成了该模块从“能用”到“好用”的跨越，还深度修补了多个因底层框架限制和 Oracle 数据库特性导致的系统级 Bug。

---

## ✅ 详细工作产出

### 1. UI/UX 像素级重构与体验升级
* **视觉语言统一**：移除原有的非标按钮（如蓝色背景的“保存计划”），全面回归系统原生的 EasyUI 简约风格，确保 UI 的全局一致性。
* **界面信噪比优化**：通过移除未加载的图标占位符（`iconCls`）和禁用冗余的折叠箭头，消除了标题面板前方的无意义留白，显著提升了首屏有效信息的展示密度。
* **录入效率极致提升**：
    * **自动聚焦 (Auto Focus)**：利用 `setTimeout` 延迟捕获弹窗动画结束的瞬间，强制将光标定位在“分组名称”或“周期名称”输入框内，实现“开窗即打字”，打断了鼠标与键盘的频繁切换。
    * **智能行复制**：新增周期行时，自动提取上一行的“周期”、“单位”、“随访方式”及“问卷模板”等高频参数，极大减少了业务人员的重复录入工作量。
* **布局对齐与空间压缩**：摒弃在复杂表单中表现不稳定的 Flex 弹性布局，全面改用**严谨的 Table 像素网格布局**。锁死 Label 宽度为 80px，严格对齐右侧输入框边缘，彻底解决了 B 端表单右边缘“参差不齐”的顽疾。
* **滚动条治理**：通过精确计算容器高度并设置 `overflow: hidden !important`，彻底消除了“外层弹窗+内层容器”的双层滚动条叠加问题，完美规避了远程桌面（RDP/VNC）操作时的滚轮事件穿透与冲突。

### 2. 功能逻辑与系统安全性增强
* **防误关二次确认**：在弹窗的“关闭”按钮及右上角关闭事件中注入拦截逻辑，检测到操作时弹出 Layer 二次确认框，有效防止用户因误触导致长表单数据丢失。
* **搜索组件性能调优**：针对私有化远程部署环境，将原本瀑布式的“所属科室”长下拉框（Combobox）降级重构为**模糊匹配文本框（Textbox）**，大幅减少了因一次性渲染海量科室数据导致的 DOM 性能开销。
* **多维查重与冲突拦截**：
    * **前端防线**：在 JS 层引入 `Set` 集合，实时遍历校验当前弹窗内是否存在“名称重复”或“时间+单位重复”的冲突配置，将错误拦截在网络请求之前。
    * **后端防线**：修复 `check周期重复` 的 SQL 语句，确保在数据库真实执行 Insert 前进行最后一道“护城河”校验，彻底杜绝违规数据入库。

## 🧠 技术成长与深度心得

### 1. 框架底层避坑：参数越界与 Layer 索引冲突
本周攻克的最隐蔽 Bug 出现在 `jserver.loadData` 的底层调用上。
* **现象还原**：在传递第 5 个参数（原本预期为 error 回调函数）时，页面触发 JS 语法报错并导致 Loading 遮罩层死锁。
* **底层原理**：查阅源码发现，该底层框架将第 5 个参数硬性解析为 `Layer` 的弹窗索引（Index）。当传入 Function 时，框架底层执行了类似 `$("#layui-layer" + function...)` 的异常拼接，导致 jQuery 选择器引擎直接崩溃（`Syntax error, unrecognized expression`）。
* **破局之道**：放弃原有的回调嵌套，利用 `Promise` 重新封装了 `jserverPromise`。不仅优雅避开了底层 API 的参数位限制，还彻底解决了批量 Save 时的异步时序（Callback Hell）问题。

### 2. Oracle 数据库适配微操
* **字段别名强制大写**：Oracle 在执行无别名引号的 `count(1) as cnt` 时，返回的 JSON 对象键名会被强制转为大写 `CNT`。在 JS 端取值时必须做好向下兼容处理：`res.obj[0].CNT || res.obj[0].cnt || 0`，防止因 `undefined` 导致前端查重逻辑被静默绕过。
* **空参 SQL 的健壮性容错**：在 XML SQL 映射中，若直接使用 `ID != :id`，当新增操作 `:id` 为空串时，极易触发 `ORA-00936: 缺失表达式` 或被框架 `nvl` 吞参。本周重构为 `(':id' Is Null Or ':id' = '' Or ID != ':id')`，显著提升了动态 SQL 在面对缺失参数时的健壮性。

| 问题现象 | 根本原因 | 解决方案 |
| :--- | :--- | :--- |

## 🏗️ 架构思考：捍卫“父子同表”的设计底气

针对业务侧关于“为何周期明细中的科室/用户字段为空，不利于报表直出”的疑问，本周在研发层面确立了以下架构共识与解释逻辑：

1.  **坚守数据强一致性 (Data Integrity)**：
    * 依据数据库**第三范式 (3NF)**，作为子表的“周期”绝不能冗余存储主表“分组”的归属属性（如科室、用户）。
    * **防脏数据推演**：若一个分组下挂载 10 个周期，且全部冗余写入了科室名称。一旦该分组发生科室调动，底层必须开启事务并发更新 11 条记录。若遇网络抖动导致部分执行失败，将直接产生“同组不同科”的致命逻辑脏数据。
2.  **物理存储与逻辑展示解耦**：
    * 物理表的第一要务是保障数据的安全与一致，而非迎合人类的直观阅读。公共属性必须“唯一收敛”在主记录中。
    * 针对业务端的“大宽表”统计需求，最佳实践是在查询层通过 `LEFT JOIN` 自连接（即 `c.分组ID = g.ID`）进行内存级展平，或在 DB 层挂载专门的只读视图（View）。
3.  **最终结论**：原表中的空值（Null）并非设计疏漏，而是保障千万级业务数据平稳流转的精密“护城河”。我们已将此架构声明固化在 `随访分组.sql.xml` 的文件头注释中，作为后续维护者的规约。

【架构与设计说明：随访分组与周期模块】
一、 存储模式：单表级联
“分组（主记录）”与“周期（子记录）”合并存储于“随访分组”物理表。
主记录标识：ID = 分组ID
子记录标识：ID != 分组ID

二、 核心设计：第三范式（3NF）与防冗余
周期记录刻意留空“所属科室”、“所属用户”等公共字段，以保障数据强一致性。
规避更新异常：若周期冗余存储科室信息，当分组发生调科或人员变动时，需同步更新所有子记录。一旦部分执行失败，将产生致命脏数据。
最佳实践：公共属性统一收敛于主记录。数据变更时仅需更新主记录，所有周期自动同步生效。

三、 研发与查询规范
物理表结构旨在保障存储安全。若业务端需“大宽表”进行直观展示或报表导出，请在 SQL 查询层处理，严禁向底层写入冗余数据。
周期数据回溯：查询周期信息时，需先自连接主记录（left join 随访分组 g on a.分组ID = g.ID），再关联字典表。
科室字段映射：本表 所属科室ID 存储为短编码。关联部门表时，必须使用 left join 部门信息 b on a.所属科室ID = b.编码。
-->

        where a.状态 != '9'

        select t.* from (
        select t.*, rownum rowno from (
        select
        a.ID, a.分组ID, a.分组名称, a.所属科室ID, b.名称 as 科室名称,
        a.所属用户ID, c.姓名 as 用户姓名, a.状态, a.创建时间, a.备注,
        (select count(1) from 随访分组 where 分组ID = a.分组ID and ID != a.分组ID and 状态 != '9') as 周期数量
        from 随访分组 a

        left join 用户信息 c on a.所属用户ID = c.ID

        and (':fzmc' Is Null Or ':fzmc' = '' Or a.分组名称 like '%' || ':fzmc' || '%')
        and (':ksmc' Is Null Or ':ksmc' = '' Or b.名称 like '%' || ':ksmc' || '%')
        order by a.创建时间 desc
        ) t
